# BUG-023 — Flood packets bypass MeshCore's reception-quality delay

[← Intentional divergence list](../../../README.md#intentional-divergences)

| Field | Value |
|---|---|
| Severity | **Not classified as a defect** |
| Area | Radio receive scheduling |
| Affected components | OpenHop Core, OpenHop Repeater |
| Status | Intentional divergence / policy under evaluation |

## TL;DR

OpenHop does not currently feed reception quality into flood forwarding in the same way as official MeshCore. RightUp has clarified that this is deliberate: OpenHop has a separately implemented, MeshCore-inspired scoring model and a neighbour-link report, while reception-quality routing remains disabled by default. The collected data is intentionally being evaluated before any score is allowed to influence flood routing.

**Current status: ⚪ Intentional divergence / policy under evaluation.** This report is no longer counted as an active defect. Official MeshCore and OpenHop both leave reception-quality routing disabled by default, but their optional strategies are not intended to be identical at this stage. OpenHop's later neighbour-link implementation records and exposes link scores without using them for forwarding, deduplication, TX delay, route selection, or packet acceptance. That observational implementation is visible in [`9688fe7` — `feat(neighbour-links): implement neighbour link tracking and history retrieval`](https://github.com/openhop-dev/openhop_repeater/commit/9688fe70e44d85f3cec7de5c77dd804d915a7506). The supplied audit snapshot predates that Repeater commit, so it is cited as maintainer context rather than used as the local source baseline.

## Original observation

Official MeshCore can delay flood processing from a reception-quality score and estimated airtime. The supplied OpenHop Core snapshot processes the first accepted flood immediately after deduplication and does not reproduce that optional queue.

## Maintainer decision

The OpenHop maintainer is deliberately collecting neighbour-link and score data before deciding whether reception quality should influence flood routing. Treating this as a required byte-for-byte implementation match would incorrectly classify a routing-policy experiment as a protocol defect.

## Compatibility assessment

The difference does not alter packet serialization, cryptographic verification, route headers, ACK formats, or companion frames. With the optional reception-quality feature disabled—the default in both systems—the absence of the same experimental routing policy is not an interoperability failure.

## Historical suggested code change

The linked file is retained only as historical context. It was generated by an LLM before the maintainer clarified that this behavior is intentional. It is **not recommended for application** and should not be treated as an outstanding fix.

## What happens

The frame-server persistence hook moves every message from the in-memory queue to SQLite. The table and reconstruction retain only basic fields and omit `snr`, `rssi`, `sender_prefix`, `channel_data_type`, and `channel_data_payload`. The hook also ignores a `False` SQLite insert result and removes the in-memory entry anyway.

## How official MeshCore handles it

Official MeshCore builds the complete companion response frame first, including the response type and fields such as SNR, sender prefix, path metadata, channel-data type, length, and binary payload. It then stores that entire frame byte-for-byte in the offline queue and later returns the same bytes.

## How the OpenHop stack handles it

**OpenHop Repeater:** A restart is not required. With SQLite enabled, the memory entry is removed immediately. A GRP_DATA message returns as empty channel text, a signed room message loses its author prefix, and signal metadata becomes zero.

## What needs to change

Migrate `companion_messages` with BLOB and numeric columns for every `QueuedMessage` field, write and read them atomically in push, pop, and load, and reconstruct the full model. Remove the memory entry only after a confirmed durable insert or explicit duplicate result; distinguish database failure from duplicate detection. Add round-trip tests for direct text, signed text, channel text, and channel data before and after restart.

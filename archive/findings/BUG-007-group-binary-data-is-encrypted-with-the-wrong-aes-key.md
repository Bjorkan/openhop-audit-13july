## TL;DR

OpenHop hashes the channel secret and uses that digest as the AES-key input for GRP_DATA. MeshCore uses the first 16 bytes of the channel secret directly. OpenHop group datagrams therefore cannot be decrypted by MeshCore peers. To fix it, use channel_secret[:16] for AES and the padded 32-byte channel secret for HMAC, matching group text and MeshCore. Remove the redundant secret arguments and add cross-language vectors.

**Current status: ✅ Fully fixed.** GRP_DATA now uses the first 16 bytes of the normalized channel secret as the AES-128 key and the complete 32-byte secret for authentication, so OpenHop and MeshCore produce compatible encrypted group datagrams. The remaining legacy `channel_secret`/`secret` parameter duplication does not alter the selected key or the wire result and is an API-cleanup concern rather than a remaining instance of this bug; see [the group-data builder](https://github.com/openhop-dev/openhop_core/blob/fix/all-the-things-core/src/openhop_core/protocol/packet_builder.py#L781-L815). The cryptographic correction was introduced in [`c4b3a85` — `fix(group): use channel secret[:16] as AES key for GRP_DATA`](https://github.com/openhop-dev/openhop_core/commit/c4b3a85). Rechecked against current Core head [`41b6201`](https://github.com/openhop-dev/openhop_core/commit/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8) and Repeater head [`dd6dfce`](https://github.com/openhop-dev/openhop_repeater/commit/dd6dfce9e89fab76967d91e202d8e47217c30474), as applicable. The original correction remains present, the changed callers preserve it, and the complete regression suites found no regression.

## What happens

OpenHop hashes the channel secret and uses that digest as the AES-key input for GRP_DATA. MeshCore uses the first 16 bytes of the channel secret directly. OpenHop group datagrams therefore cannot be decrypted by MeshCore peers.

## How official MeshCore handles it

Mesh::createGroupDatagram passes channel.secret to encryptThenMAC. Utils::encrypt uses the first CIPHER_KEY_SIZE bytes for AES-128, while HMAC uses all 32 secret bytes.

## How the OpenHop stack handles it

**OpenHop Core:** create_group_data_packet computes SHA-256(secret) and passes that as the encryption key, even though its group-text builder correctly uses secret[:16].

## What needs to change

Use channel_secret[:16] for AES and the padded 32-byte channel secret for HMAC, matching group text and MeshCore. Remove the redundant secret arguments and add cross-language vectors.

## TL;DR

OpenHop originally omitted MeshCore’s optional reception-quality hold and deliberately collected neighbour-score data before enabling routing effects. The current snapshots now implement the official score-based flood hold while retaining the separate observational neighbour-link report. The feature remains disabled by default, matching firmware defaults.

**Current status: ✅ Fully implemented.** The former intentional divergence is no longer present. Core now implements MeshCore’s packet-score calculation, RadioLib-compatible airtime estimate, `calcRxDelay` exponent, 50 ms threshold, 32-second cap, flood-only hold, and—critically—the deduplication check after the hold; see [`dispatcher.py` L390–L520](https://github.com/openhop-dev/openhop_core/blob/abe1d857734d5d1f549ada27af6f95c7e5e53e95/src/openhop_core/node/dispatcher.py#L390-L520) and [`packet_utils.py` L46–L113](https://github.com/openhop-dev/openhop_core/blob/abe1d857734d5d1f549ada27af6f95c7e5e53e95/src/openhop_core/protocol/packet_utils.py#L46-L113). Repeater wires the disabled-by-default `rx_delay_base` setting into the dispatcher and supports live reload. The observational neighbour-link report remains separate and bounded. Implemented across Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519). Rechecked against latest supplied Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519), as applicable. The original correction remains present; the full test suites and changed-path review found no regression.

## Original observation

Official MeshCore can delay flood processing from a reception-quality score and estimated airtime. The supplied OpenHop Core snapshot processes the first accepted flood immediately after deduplication and does not reproduce that optional queue.

## Resolution history

RightUp initially deferred routing changes while neighbour-link data was collected. The later Core and Repeater commits implemented MeshCore’s reception-quality hold directly, so the report is now archived as fully implemented rather than as an intentional divergence.

## Compatibility assessment

The optional behavior now matches the relevant MeshCore receive-order semantics: delay is calculated only for flood routes, remains disabled at the default base value, and deduplication happens after the hold. Packet serialization and cryptographic behavior are unchanged.

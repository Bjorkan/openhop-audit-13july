## TL;DR

OpenHop records invalid UTF-8 name bytes only as diagnostic metadata, then the advert handler rejects the contact because no decoded name is present. MeshCore copies the bounded name bytes as C text without requiring valid UTF-8. To fix it, preserve the wire bytes and decode only for display with a reversible or replacement policy. Parsing must not discard the advert merely because its display name is not valid UTF-8.

**Current status: ✅ Fully fixed.** Invalid UTF-8 no longer invalidates an otherwise authenticated advert: the parser preserves the raw name bytes and produces replacement-decoded display text, allowing the contact to continue through the advert pipeline. ## What happens

OpenHop records invalid UTF-8 name bytes only as diagnostic metadata, then the advert handler rejects the contact because no decoded name is present. MeshCore copies the bounded name bytes as C text without requiring valid UTF-8.

## How official MeshCore handles it

MeshCore copies and terminates the advertised name without UTF-8 validation; protocol routing and contact storage do not depend on Unicode validity.

## How the OpenHop stack handles it

**OpenHop Core:** The decoder catches `UnicodeDecodeError` and stores `raw_name_bytes`, but it does not produce `node_name`. `AdvertHandler` then treats the advert as nameless and returns without storing or publishing the contact.

## What needs to change

Preserve the wire bytes and decode only for display with a reversible or replacement policy. Parsing must not discard the advert merely because its display name is not valid UTF-8.

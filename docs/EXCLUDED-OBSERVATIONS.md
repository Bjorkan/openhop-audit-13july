# Excluded Observations

[← Audit home](../README.md)

The observations below were evaluated but are not open defects under the declared compatibility profile. They are included to make the audit boundary explicit.

| ID | Classification | Observation | Assessment |
|---|---|---|---|
| BUG-001 | Intentional forward-compatible parsing | Reserved payload version 1 is accepted | OpenHop intentionally accepts reserved payload version 1 while rejecting higher values. This is a permissive parser choice and does not alter valid version-0 traffic. |
| BUG-004 | Resolved by the integrated stack | The exported cipher-MAC width contradicts the implementation | The public constant is inconsistent, but the active cryptographic path uses two bytes and the combined core and repeater produce the correct wire format. This is API cleanup rather than an open stack defect. |
| BUG-005 | Not a product defect | Packet hashes use 32 bytes instead of MeshCore's 8 bytes | OpenHop uses the full SHA-256 value internally and truncates to eight bytes where a MeshCore packet identifier requires it. A wider internal key is not inherently incompatible, and no incorrect wire data was demonstrated. |
| BUG-006 | Not a functional stack defect | Identity addresses are derived from the wrong bytes | Repeater routing, local hash selection, and handler lookup use the public-key prefix directly. The hashed `get_address_bytes()` representation appears in diagnostics and REST output but does not control MeshCore routing or encryption. |
| BUG-014 | No independent defect | Ordinary text messages encrypt an extra NUL byte | The extra NUL produces non-canonical and sometimes longer ciphertext, but MeshCore receivers accept zero padding. Practical length risk is covered by BUG-013 and receive parsing by BUG-016. |
| BUG-019 | Unsupported virtual-node capability | MeshCore location policy value 2 is unsupported | A repeater-hosted companion identity is a virtual node and does not implement MeshCore firmware location-policy value 2. Repeater location is owned by daemon configuration and GPS services instead. |
| BUG-021 | Intentional defensive normalization | Future advert timestamps are rewritten | Clamping future advert timestamps to local time is a policy difference, but no independent interoperability failure was established in the integrated stack. |
| BUG-027 | Resolved by the integrated stack | Encrypted PATH ACKs are ignored unless they are exactly 20 bytes and a waiter exists | The current core and repeater paths handle the relevant PATH-ACK variants. |
| BUG-030 | No meaningful compatibility defect | A host-supplied text timestamp of zero is rewritten | Zero is used as a sentinel for automatic current time in this API path. Normal companion clients do not lose a valid message or ACK because of this behavior. |
| BUG-031 | Permissive forward compatibility | Unsupported direct-text types are delivered | Unknown direct-text types are accepted rather than rejected. Supported types are unaffected; the difference applies only to constructed or unknown input. |
| BUG-032 | Resolved by the integrated stack | ModemIdentity cannot satisfy the core encryption call pattern | The repeater uses an identity path that satisfies the encryption call pattern, so the standalone ModemIdentity limitation does not affect the reviewed stack. |
| BUG-033 | Resolved by the integrated stack | Received flood packets are never forwarded | RepeaterHandler implements flood forwarding and appends the local path hash. |
| BUG-034 | Resolved by the integrated stack | Directed packets are not forwarded to their next hop | RepeaterHandler implements direct forwarding and removes the next hop. |
| BUG-035 | Resolved by the integrated stack | TRACE route semantics and forwarding are not implemented | TraceHelper implements TRACE validation, next-hop matching, SNR append, and forwarding. |
| BUG-037 | Resolved by the integrated stack | Own-packet detection reads unrelated payload bytes | The repeater replaces the unsuitable own-packet heuristic and uses its own dedupe and `mark_seen` path. |
| BUG-038 | Resolved by the integrated stack | Outgoing packets are not registered in the receive dedupe table | Locally transmitted packets are registered in the repeater seen cache and Dispatcher filter. |
| BUG-039 | Implementation choice without demonstrated defect | Deduplication expires by 30-second time instead of a 160-packet cycle | MeshCore uses a 160-entry cycle while OpenHop uses time- and size-bounded caches. No protocol rule requires the exact eviction algorithm, and no separate incorrect wire or routing event was demonstrated. |
| BUG-041 | Overbroad characterization | The MeshCore transmit scheduler is replaced by an immediate lock | The repeater includes delay, duty-cycle control, listen-before-talk, serialized radio access, and shutdown draining. It does not copy MeshCore queue architecture exactly, but it is not an immediate-lock-only transmitter. Concrete timing defects remain in BUG-040, BUG-084, and BUG-086. |
| BUG-042 | Resolved by the integrated stack | Protocol requests cannot produce a transmitted response | ProtocolRequestHelper builds responses and injects them through the repeater packet injector. |
| BUG-044 | Resolved by the integrated stack | Anonymous server requests are never dispatched to AnonRequestHandler | PacketRouter and LoginHelper dispatch anonymous requests through the active login-server path. |
| BUG-045 | Resolved by the integrated stack | Login replay protection is optional application behavior | The repeater ACL rejects `timestamp <= client.last_timestamp` after successful authentication, so replay checking is active in the production login path. |
| BUG-047 | Resolved by the integrated stack | TRACE callbacks are selected from the last hop byte instead of the request tag | TraceHelper correlates pending pings with the TRACE tag. |
| BUG-048 | Resolved by the integrated stack | Every TRACE result is delivered to every protocol-response waiter | TRACE results use a separate TraceHelper callback path rather than being delivered to every protocol-response waiter. |
| BUG-052 | Intentional reliability extension | OpenHop retransmits high-level requests up to three times | The high-level OpenHop request API intentionally retries transiently unanswered requests. This differs from firmware command behavior but is an application-layer reliability policy, not a wire-format defect. |
| BUG-053 | Unsupported characterization removed during verification | Transport region policy is incomplete outside scoped flood forwarding | The supplied MeshCore code applies the reviewed region forwarding denial to route-flood packets; `REGION_DENY_DIRECT` is reserved for future use. The original report therefore did not demonstrate a current official-behavior mismatch. |
| BUG-054 | Duplicate root cause | Encrypted peer handlers do not require the packet to be addressed locally | The concrete risks are premature payload dispatch and one-byte destination collisions. They are covered by BUG-089 and BUG-098. |
| BUG-057 | Declared input constraint | Non-ASCII region names are not supported | The reviewed repeater compatibility profile uses ASCII region identifiers such as hashtag or IATA-style scope names. Non-ASCII region names are outside that profile. |
| BUG-071 | Intentionally unsupported administrative capabilities | Five MeshCore companion commands are outside the supported virtual-companion profile | REBOOT, FACTORY_RESET, SET_DEVICE_PIN, HAS_CONNECTION, and GET_TUNING_PARAMS are not part of the supported virtual-companion capability set. A companion connection must not administer the host operating system or global repeater state. Unsupported commands must continue to return an explicit unsupported response. |
| BUG-072 | Merged into the radio-command capability finding | SET_RADIO_PARAMS differs from dedicated companion-radio firmware | SET_RADIO_PARAMS is intentionally not allowed to mutate the shared repeater radio. Frequency and client-repeat validation are therefore not separate defects; the remaining issue is false success and misleading virtual preference mutation, covered by BUG-090. |
| BUG-073 | Unsupported client-repeat capability | GET_ALLOWED_REPEAT_FREQ returns an empty capability list | The shared single-radio repeater architecture does not expose client-repeat frequency ranges to virtual companion identities. An empty capability list is valid for this profile. |
| BUG-075 | Intentional OpenHop extension | Idle TCP connections receive unsolicited current-time responses | OpenHop explicitly documents and implements `RESP_CODE_CURR_TIME` as its TCP heartbeat. It can surprise strict clients, but it is intentional and is not retained as an unintended defect. |
| BUG-077 | Unsupported virtual-node telemetry capability | The self-telemetry command form is unsupported | A virtual companion identity has no independent battery or sensor set. Local repeater telemetry is exposed through repeater services rather than the firmware self-telemetry command. |
| BUG-078 | Intentional protocol extension | GET_CHANNEL includes OpenHop extension forms | The no-index full-list form and blank valid-slot response are OpenHop companion extensions. Valid indexed MeshCore requests remain supported; callers requiring strict firmware semantics should not use the extension form. |
| BUG-079 | Allowed parser extension | SET_CHANNEL accepts key formats MeshCore explicitly rejects | Accepting additional key input forms does not make the valid 16-byte form incorrect. The behavior is more permissive than firmware but does not break supported input. |
| BUG-081 | Merged into the radio-command capability finding | TX-power mutation validation does not use a hardware maximum | Virtual companions are not permitted to set global TX power. Target-specific mutation validation is therefore not a separate defect. Read-only maximum-power reporting remains covered by BUG-068, and false success is covered by BUG-090. |
| BUG-082 | Malformed-input difference only | GET_STATS without a subtype silently defaults to packet stats | A valid GET_STATS command includes a subtype. Default behavior for a missing subtype affects only an invalid frame and is not an independent product defect. |

## What happens

The TCP frame server closes a client after eight hours without inbound command bytes, even if the connection is healthy and continues to receive mesh pushes. Official companion firmware does not impose an application-command idle deadline. OpenHop’s own docstring acknowledges that None matches firmware, but both Core and Repeater default to 28,800 seconds.

## How official MeshCore handles it

The companion firmware enables its serial interface and keeps it available. It does not disconnect an application because no command frame was sent for a fixed period; physical/link-layer connection management owns liveness.

## How the OpenHop stack handles it

CompanionFrameServer defaults client_idle_timeout_sec to eight hours. The read loop wraps reader.read(1) in wait_for() and returns idle_timeout on expiry. Outbound pushes and the fabricated heartbeat do not reset that inbound-only deadline.

## What needs to change

Default the idle timeout to None for firmware-compatible behavior. Deployments that need slot reclamation can opt into a timeout or use TCP keepalive and socket failure detection. Add a test where a client receives pushes for longer than the configured period without sending commands and remains connected by default.

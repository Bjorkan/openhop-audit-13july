# BUG-087 — Short companion commands are accepted and can silently mutate device state

## TL;DR

OpenHop accepts zero-length payloads for state-changing commands that official MeshCore only handles when required bytes are present. An empty SET_ADVERT_NAME clears the node name, and an empty SET_FLOOD_SCOPE_KEY resets the scope override while returning OK. Empty GET_STATS also produces a default packet-statistics response instead of being unsupported.

**Current status: 🔴 Not fixed.** Confirmed in the supplied Core snapshot [`9ea7269`](https://github.com/openhop-dev/openhop_core/commit/9ea7269a7e7e903fe433b1f952a4026fe3dcc81b) and Repeater snapshot [`6aafa7f`](https://github.com/openhop-dev/openhop_repeater/commit/6aafa7fe991b5b3199b18149f84417f8522d94b2), as applicable. ## What happens

OpenHop accepts zero-length payloads for state-changing commands that official MeshCore only handles when required bytes are present. An empty SET_ADVERT_NAME clears the node name, and an empty SET_FLOOD_SCOPE_KEY resets the scope override while returning OK. Empty GET_STATS also produces a default packet-statistics response instead of being unsupported.

## How official MeshCore handles it

Firmware dispatch guards require at least one name byte for SET_ADVERT_NAME, a mode byte for SET_FLOOD_SCOPE_KEY, and a statistics subtype for GET_STATS. Short frames fall through to UNSUPPORTED_CMD.

## How the OpenHop stack handles it

The handlers synthesize defaults from absent data and report success, turning malformed or truncated frames into valid state changes.

## What needs to change

Enforce each official minimum frame length before reading defaults. Return the standard unsupported/illegal response without changing state, and add truncated-frame tests for all command handlers.

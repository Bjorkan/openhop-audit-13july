## TL;DR

OpenHop accepts 16-byte channel secrets, raw 32-byte secrets, and 64-character hexadecimal secrets on the standard companion command. Official MeshCore explicitly rejects the 32-byte command form and supports only the 16-byte wire secret. A companion can therefore save channel state through OpenHop that the advertised firmware protocol refuses.

## What happens

OpenHop accepts 16-byte channel secrets, raw 32-byte secrets, and 64-character hexadecimal secrets on the standard companion command. Official MeshCore explicitly rejects the 32-byte command form and supports only the 16-byte wire secret. A companion can therefore save channel state through OpenHop that the advertised firmware protocol refuses.

## How official MeshCore handles it

The 32-byte SET_CHANNEL frame branch returns UNSUPPORTED_CMD. The supported branch copies exactly 16 secret bytes into the channel key.

## How the OpenHop stack handles it

The handler chooses among three encodings by frame length and normalizes all of them into the channel store.

## What needs to change

Keep any extended secret normalization in a separate host API. On the MeshCore companion command, accept only the 16-byte form and return UNSUPPORTED_CMD for the official 32-byte form; reject ambiguous trailing encodings.

## TL;DR

`send_repeater_command()` installs one global callback on `TextMessageHandler`. While it is active, the next successfully decrypted direct text message—regardless of `txt_type` or sender—is consumed as the command reply. Concurrent commands overwrite each other’s callback, and an ordinary chat message or another repeater’s CLI reply can complete the wrong request and disappear from normal message delivery.

## What happens

`send_repeater_command()` installs one global callback on `TextMessageHandler`. While it is active, the next successfully decrypted direct text message—regardless of `txt_type` or sender—is consumed as the command reply. Concurrent commands overwrite each other’s callback, and an ordinary chat message or another repeater’s CLI reply can complete the wrong request and disappear from normal message delivery.

## How official MeshCore handles it

MeshCore dispatches `TXT_TYPE_CLI_DATA` through `onCommandDataRecv()` while plain and signed messages use separate callbacks. The contact object remains part of the callback, allowing callers to correlate the reply with the intended peer.

## How the OpenHop stack handles it

The text handler checks only whether `command_response_callback` is non-null. The callback in `send_repeater_command()` ignores its `sender_contact` argument, and each call replaces the handler-wide callback until its `finally` block clears it.

## What needs to change

Replace the global callback with request-scoped waiters keyed by the full target public key, and dispatch only `TXT_TYPE_CLI_DATA` to them. Protect registration/removal from concurrent calls, leave unmatched messages on the normal delivery path, and test interleaved commands plus intervening plain messages.

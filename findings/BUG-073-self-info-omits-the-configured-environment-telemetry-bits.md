## TL;DR

OpenHop stores the companion environment setting but omits it from the packed `SELF_INFO` flags. Clients always observe environment mode zero after APP_START even when another value is configured.

**Current status: 🔴 Not fixed.** Confirmed in the supplied Core snapshot [`9ea7269`](https://github.com/openhop-dev/openhop_core/commit/9ea7269a7e7e903fe433b1f952a4026fe3dcc81b) and Repeater snapshot [`6aafa7f`](https://github.com/openhop-dev/openhop_repeater/commit/6aafa7fe991b5b3199b18149f84417f8522d94b2), as applicable. The full test suites pass, but no focused implementation or regression test resolves this compatibility difference.

## What happens

OpenHop stores the companion environment setting but omits it from the packed `SELF_INFO` flags. Clients always observe environment mode zero after APP_START even when another value is configured.

## How official MeshCore handles it

The firmware packs the environment value into bits 4–5 of the SELF_INFO flags byte alongside manual-add and telemetry mode.

## How the OpenHop stack handles it

The response builds only the manual-add and telemetry bits. The environment preference is available in `NodePrefs` but not ORed into the byte.

## What needs to change

Pack `(environment & 0x03) << 4` in the response and add round-trip tests for all four environment values.

# BUG-071 — Repeater status hardcodes TX queue depth and error events to zero

## TL;DR

The status response advertises MeshCore-compatible `RepeaterStats` fields but always serializes zero for current TX queue length and error events. Monitoring clients cannot distinguish an idle healthy repeater from one with a backed-up queue or recorded faults.

**Current status: 🔴 Not fixed.** Confirmed in the supplied Core snapshot [`9ea7269`](https://github.com/openhop-dev/openhop_core/commit/9ea7269a7e7e903fe433b1f952a4026fe3dcc81b) and Repeater snapshot [`6aafa7f`](https://github.com/openhop-dev/openhop_repeater/commit/6aafa7fe991b5b3199b18149f84417f8522d94b2), as applicable. The full test suites pass, but no focused implementation or regression test resolves this compatibility difference.

## What happens

The status response advertises MeshCore-compatible `RepeaterStats` fields but always serializes zero for current TX queue length and error events. Monitoring clients cannot distinguish an idle healthy repeater from one with a backed-up queue or recorded faults.

## How official MeshCore handles it

The firmware reports the outbound manager's current queue total and its accumulated error flags.

## How the OpenHop stack handles it

The response builder includes explicit TODO/hardcoded zero values despite the Repeater stack having a bounded packet queue and observable runtime failures.

## What needs to change

Expose a concurrency-safe queue-depth accessor and a defined error-bit accumulator, then serialize those values. Document any host-only error-bit mapping.

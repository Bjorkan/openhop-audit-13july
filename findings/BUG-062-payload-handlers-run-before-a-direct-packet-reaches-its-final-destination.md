# BUG-062 — Payload handlers run before a direct packet reaches its final destination

[← Bug list](../README.md#bug-list)

| Field | Value |
|---|---|
| Severity | **High** |
| Area | Repeater routing order |
| Affected components | OpenHop Repeater |
| Status | Confirmed from the supplied source snapshots |

## TL;DR

PacketRouter runs advert, text, PATH, request, group, and companion handlers before `RepeaterHandler.direct_forward`. It does not first check whether a direct packet still has routing hops remaining. To fix it, place the MeshCore route-state machine before payload dispatch. For direct packets with `hop_count > 0`, run only TRACE, the CONTROL rule, early ACK, and multipart special cases; send all other packets directly to next-hop forwarding.

## What happens

PacketRouter runs advert, text, PATH, request, group, and companion handlers before `RepeaterHandler.direct_forward`. It does not first check whether a direct packet still has routing hops remaining.

## How official MeshCore handles it

MeshCore handles TRACE, CONTROL, and early ACK cases specially, then returns from the direct path-forwarding branch when hops remain. Payload dispatch and local decryption occur only when the direct path count reaches zero.

## How the OpenHop stack handles it

**OpenHop Repeater:** An intermediate repeater can store an advert, deliver group text, or attempt to consume encrypted traffic intended for the final node. A local one-byte hash collision can also set `processed_by_injection` and stop required forwarding.

## What needs to change

Place the MeshCore route-state machine before payload dispatch. For direct packets with `hop_count > 0`, run only TRACE, the CONTROL rule, early ACK, and multipart special cases; send all other packets directly to next-hop forwarding. Run local handlers only at final direct hop or in the flood-local processing branch.

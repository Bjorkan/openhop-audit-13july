# BUG-123 — CMD_RESET_PATH does not persist the cleared route

[← Active finding list](../README.md#active-findings)

## TL;DR

CMD_RESET_PATH clears a contact route in the in-memory ContactStore but never invokes the persistence hook used by add, update and remove commands. On a SQLite-backed Repeater companion, restarting can restore the old direct path and undo the command. Official firmware marks the contact store dirty after resetting the path.

**Current status: 🔴 Not fixed.** Rechecked against latest supplied Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519), as applicable. The three new Core commits and two new Repeater commits do not remove this mismatch; the complete regression suites and focused changed-path review found no contrary behavior.

## What happens

CMD_RESET_PATH clears a contact route in the in-memory ContactStore but never invokes the persistence hook used by add, update and remove commands. On a SQLite-backed Repeater companion, restarting can restore the old direct path and undo the command. Official firmware marks the contact store dirty after resetting the path.

## How official MeshCore handles it

The companion firmware sets `out_path_len` to unknown and schedules the lazy contact-store write before returning OK.

## How the OpenHop stack handles it

The command calls `bridge.reset_path()` and immediately writes OK. Adjacent add/update and remove handlers await `_save_contacts()`, but reset-path does not.

## What needs to change

Persist the contact set after a successful reset, preferably through one shared mutation helper so add/update/remove/reset cannot diverge. If persistence fails, define whether the command rolls back or returns an error rather than acknowledging a non-durable state. Test restart behavior with the SQLite bridge.

## TL;DR

OpenHop exports `CIPHER_MAC_SIZE = 32`, while the only supported MeshCore wire version uses a two-byte MAC. The internal crypto paths often hardcode the correct two-byte truncation, so ordinary traffic can still work, but integrations that consume the public constant calculate the wrong wire overhead or parse the wrong number of bytes.

**Current status: 🔴 Not fixed.** Confirmed in the supplied Core snapshot [`9ea7269`](https://github.com/openhop-dev/openhop_core/commit/9ea7269a7e7e903fe433b1f952a4026fe3dcc81b) and Repeater snapshot [`6aafa7f`](https://github.com/openhop-dev/openhop_repeater/commit/6aafa7fe991b5b3199b18149f84417f8522d94b2), as applicable. The full test suites pass, but no focused implementation or regression test resolves this compatibility difference.

## What happens

OpenHop exports `CIPHER_MAC_SIZE = 32`, while the only supported MeshCore wire version uses a two-byte MAC. The internal crypto paths often hardcode the correct two-byte truncation, so ordinary traffic can still work, but integrations that consume the public constant calculate the wrong wire overhead or parse the wrong number of bytes.

## How official MeshCore handles it

MeshCore defines `CIPHER_MAC_SIZE` as 2 for payload version 1. This is the authenticated tag length carried by encrypted direct and group payloads.

## How the OpenHop stack handles it

The public protocol constant says 32 and describes the full SHA-256 HMAC rather than the transmitted truncation. That disagrees with the wire format exposed by the same package.

## What needs to change

Change the public wire constant to 2. If a full digest length is useful internally, expose it under a separate name and add tests that calculate encrypted payload overhead from the wire constants.

# BUG-069 — HAS_CONNECTION and GET_TUNING_PARAMS are advertised but unimplemented

## TL;DR

The companion constants expose the official command numbers for `CMD_HAS_CONNECTION` and `CMD_GET_TUNING_PARAMS`, but the frame-server dispatch registry has no handlers for either command. Compatible clients receive the generic unsupported-command response instead of the firmware-defined data.

**Current status: 🔴 Not fixed.** Confirmed in the supplied Core snapshot [`9ea7269`](https://github.com/openhop-dev/openhop_core/commit/9ea7269a7e7e903fe433b1f952a4026fe3dcc81b) and Repeater snapshot [`6aafa7f`](https://github.com/openhop-dev/openhop_repeater/commit/6aafa7fe991b5b3199b18149f84417f8522d94b2), as applicable. The full test suites pass, but no focused implementation or regression test resolves this compatibility difference.

## What happens

The companion constants expose the official command numbers for `CMD_HAS_CONNECTION` and `CMD_GET_TUNING_PARAMS`, but the frame-server dispatch registry has no handlers for either command. Compatible clients receive the generic unsupported-command response instead of the firmware-defined data.

## How official MeshCore handles it

The companion firmware returns current tuning parameters and reports whether a requested public key has an active connection/path.

## How the OpenHop stack handles it

Both constants are exported, and Core already has tuning-parameter state, but neither command is registered in the command map.

## What needs to change

Implement and register both commands with the exact response frame layouts used by firmware. Add frame-level tests covering positive and negative connection results and all tuning fields.

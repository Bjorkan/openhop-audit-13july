# BUG-121 — Local identities can use MeshCore-reserved 0x00 and 0xFF prefixes

## TL;DR

OpenHop accepts generated and configured local identities whose first public-key byte is `0x00` or `0xFF`. Official MeshCore treats those one-byte identity hashes as reserved and regenerates new identities. Because OpenHop routing, local identity lookup and companion persistence use that first byte, a reserved prefix can collide with protocol sentinel semantics or produce an identity that official nodes deliberately never create.

## What happens

OpenHop accepts generated and configured local identities whose first public-key byte is `0x00` or `0xFF`. Official MeshCore treats those one-byte identity hashes as reserved and regenerates new identities. Because OpenHop routing, local identity lookup and companion persistence use that first byte, a reserved prefix can collide with protocol sentinel semantics or produce an identity that official nodes deliberately never create.

## How official MeshCore handles it

Both companion and repeater firmware regenerate a newly created keypair while the first public-key byte is `0x00` or `0xFF` (with a bounded retry loop) before saving it.

## How the OpenHop stack handles it

`LocalIdentity` generates or imports a key without checking the public-key prefix. Repeater `IdentityManager` rejects duplicate prefixes but does not reject the two reserved values.

## What needs to change

Centralize local-identity validation. Generated keys should retry until the prefix is non-reserved; imported/configured identities should fail closed with a clear configuration error rather than silently replacing user key material. Apply the same validation before registering every repeater, room and companion identity, with deterministic tests for both reserved values.

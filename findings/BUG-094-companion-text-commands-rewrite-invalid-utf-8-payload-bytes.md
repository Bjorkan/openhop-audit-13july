## TL;DR

The companion wire format passes message text as bytes. Official MeshCore copies those bytes without UTF-8 validation. OpenHop decodes with `errors='replace'` and later re-encodes, so every invalid byte sequence is replaced by the three-byte U+FFFD encoding and the transmitted text, length, packet hash and ACK all change.

## What happens

The companion wire format passes message text as bytes. Official MeshCore copies those bytes without UTF-8 validation. OpenHop decodes with `errors='replace'` and later re-encodes, so every invalid byte sequence is replaced by the three-byte U+FFFD encoding and the transmitted text, length, packet hash and ACK all change.

## How official MeshCore handles it

For direct messages firmware null-terminates the incoming buffer and uses the C byte string; channel messages are copied with the explicit frame length. Neither path rewrites invalid UTF-8 bytes.

## How the OpenHop stack handles it

Both direct and channel command handlers perform replacement decoding before calling string-based send APIs. The original wire bytes cannot be recovered after this conversion.

## What needs to change

Keep companion text as bytes through packet composition, or use a reversible representation such as surrogateescape and re-encode exactly. Apply null-termination behavior only where firmware uses a C string, and test invalid byte sequences and embedded NULs separately for direct and channel commands.

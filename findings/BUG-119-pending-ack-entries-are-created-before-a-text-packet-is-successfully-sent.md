# BUG-119 — Pending ACK entries are created before a text packet is successfully sent

[← Active finding list](../README.md#active-findings)

| Field | Value |
|---|---|
| Severity | **High** |
| Area | Companion ACK lifecycle |
| Affected components | OpenHop Core |

## TL;DR

OpenHop inserts an expected ACK into its pending table before `_send_packet()` reports whether the packet was transmitted. A failed or exceptional send therefore leaves a phantom pending ACK. That stale entry can consume bounded table capacity, evict a legitimate pending send, or later turn a colliding ACK CRC into a false SEND_CONFIRMED event. Official companion firmware adds the ACK only after `sendMessage()` returned a non-failure result.

**Current status: 🔴 Not fixed.** Rechecked against latest supplied Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519), as applicable. The three new Core commits and two new Repeater commits do not remove this mismatch; the complete regression suites and focused changed-path review found no contrary behavior.

## What happens

OpenHop inserts an expected ACK into its pending table before `_send_packet()` reports whether the packet was transmitted. A failed or exceptional send therefore leaves a phantom pending ACK. That stale entry can consume bounded table capacity, evict a legitimate pending send, or later turn a colliding ACK CRC into a false SEND_CONFIRMED event. Official companion firmware adds the ACK only after `sendMessage()` returned a non-failure result.

## How official MeshCore handles it

The companion firmware calls `sendMessage()`, handles `MSG_SEND_FAILED`, and enters `expected_ack` in the circular table only in the successful branch. A failed send cannot create ACK state.

## How the OpenHop stack handles it

`send_text_message()` calls `_track_pending_ack(ack_crc)` before either the acknowledged or unacknowledged `_send_packet()` call. Neither the false-result path nor the exception path removes that entry.

## What needs to change

Create pending ACK state only after `_send_packet()` succeeds. For callers that must register before an immediate ACK can arrive, make registration and send an explicit transaction: provision a cancellable entry, transmit, then remove it on every failure/exception path. Add regression tests for false send results, thrown send errors, capacity eviction and late CRC collisions.

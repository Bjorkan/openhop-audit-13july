## What happens

When delivering a stored room post, OpenHop generates a fresh wall-clock timestamp and fixes the two attempt bits to zero. Official MeshCore sends the original stored post timestamp and randomizes the low attempt bits for each push. OpenHop clients therefore see delivery time instead of post time, and retries can recreate the same plaintext, packet hash and expected ACK within the same second.

## How official MeshCore handles it

Firmware copies `post.post_timestamp`, generates a fresh random byte, and places its low two bits into the signed-text flags before calculating the ACK.

## How the OpenHop stack handles it

The push path calls `time.time()` and uses only `TXT_TYPE_SIGNED_PLAIN << 2`; `post['post_timestamp']` is used later for sync state but not serialized into the message.

## What needs to change

Serialize the stored post timestamp and draw fresh attempt bits for every transmission attempt. Calculate the expected ACK from that exact plaintext and retain the same post timestamp as the sync watermark.

# BUG-104 — CMD_ADD_UPDATE_CONTACT rejects the valid 35-byte minimum body

## TL;DR

The frame server removes the command byte before calling `_cmd_add_update_contact()`, but the handler still requires at least 36 body bytes. Official MeshCore accepts a 36-byte command frame total—one command byte plus a 35-byte body containing the public key, type, flags and path-length byte. ## What happens

The frame server removes the command byte before calling `_cmd_add_update_contact()`, but the handler still requires at least 36 body bytes. Official MeshCore accepts a 36-byte command frame total—one command byte plus a 35-byte body containing the public key, type, flags and path-length byte. A minimal valid firmware-compatible update is therefore rejected by OpenHop as `ILLEGAL_ARG`.

## How official MeshCore handles it

The companion firmware dispatch condition is `len >= 1 + 32 + 2 + 1`. Its helper treats the remaining contact fields as extended data, with a current-time fallback for missing `lastmod`.

## How the OpenHop stack handles it

The common frame server passes `frame[1:]` to command handlers. `_cmd_add_update_contact()` then checks `len(data) < 36`, one byte too many, even though its mandatory field reads stop at body offset 34.

## What needs to change

Change the minimum body length to 35 and retain zero/default handling for omitted fixed/optional fields. Add tests for exactly 35 body bytes, 34 bytes, the full legacy body and bodies carrying the optional GPS/lastmod fields.

## TL;DR

OpenHop labels its authenticated request response delay as matching C++ but defines it as 500 ms. MeshCore’s companion, sensor, room-server and repeater implementations all use 300 ms. OpenHop responses are therefore systematically 200 ms later than firmware, affecting request latency and collision timing.

## What happens

OpenHop labels its authenticated request response delay as matching C++ but defines it as 500 ms. MeshCore’s companion, sensor, room-server and repeater implementations all use 300 ms. OpenHop responses are therefore systematically 200 ms later than firmware, affecting request latency and collision timing.

## How official MeshCore handles it

`SERVER_RESPONSE_DELAY` is 300 ms in `BaseChatMesh` and in the official server examples, and is used for direct, flood and reciprocal-PATH responses.

## How the OpenHop stack handles it

Core defines `SERVER_RESPONSE_DELAY_MS = 500`; Repeater imports it and sleeps for that interval before injecting every authenticated protocol response.

## What needs to change

Set the shared constant to 300 ms and keep any product-specific additional scheduling separate. Add timing tests around direct, flood and PATH response injection using a fake monotonic clock.

# BUG-128 — A companion TCP connection removes every host event subscriber

[← Active finding list](../README.md#active-findings)

## TL;DR

When a TCP companion connects, FrameServer calls bridge.clear_push_callbacks(), which clears the same public callback lists used by the embedding application. Any message, advert, telemetry, login, trace, or other subscriber registered before the connection is silently removed. The transport then installs only its own callbacks, changing host behavior merely because a client connected.

## What happens

When a TCP companion connects, FrameServer calls bridge.clear_push_callbacks(), which clears the same public callback lists used by the embedding application. Any message, advert, telemetry, login, trace, or other subscriber registered before the connection is silently removed. The transport then installs only its own callbacks, changing host behavior merely because a client connected.

## How official MeshCore handles it

Official firmware owns its serial interface directly and does not have a shared host callback registry that a client connection can erase. A host transport adapter must scope and remove only subscriptions it created.

## How the OpenHop stack handles it

clear_push_callbacks() empties every list in _push_callbacks. _setup_push_callbacks() calls it on every connection, and _handle_client() runs that setup after accepting the socket. A focused check registered an application callback and confirmed that it disappeared.

## What needs to change

Return subscription handles or add owner-scoped callback registries. FrameServer should detach only callbacks from its previous client binding, never clear unrelated application callbacks. Add reconnect tests with persistent host subscribers and multiple frame servers sharing one bridge.

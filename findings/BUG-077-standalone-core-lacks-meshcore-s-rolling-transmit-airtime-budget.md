# BUG-077 — Standalone Core lacks MeshCore’s rolling transmit-airtime budget

## TL;DR

The generic Core dispatcher serializes radio sends but does not implement MeshCore's rolling airtime budget. Standalone MeshNode and CompanionRadio users can therefore queue traffic faster than official firmware permits, even when a particular radio wrapper performs channel activity detection.

**Current status: 🔴 Not fixed.** Rechecked against current Core head [`41b6201`](https://github.com/openhop-dev/openhop_core/commit/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8) and Repeater head [`dd6dfce`](https://github.com/openhop-dev/openhop_repeater/commit/dd6dfce9e89fab76967d91e202d8e47217c30474), as applicable. The mismatch remains present; the new commit range does not complete this MeshCore compatibility path.

## What happens

The generic Core dispatcher serializes radio sends but does not implement MeshCore's rolling airtime budget. Standalone MeshNode and CompanionRadio users can therefore queue traffic faster than official firmware permits, even when a particular radio wrapper performs channel activity detection.

## How official MeshCore handles it

The firmware dispatcher initializes, replenishes and debits a transmit airtime budget and gates outbound selection on the available budget in addition to CAD/channel checks.

## How the OpenHop stack handles it

The dispatcher has a TX lock and ACK timing but no rolling budget derived from transmitted airtime. Repeater's separate AirTimeManager does not protect direct Core users.

## What needs to change

Add a dispatcher-level budget matching firmware semantics, based on actual packet airtime and elapsed monotonic time. Ensure Repeater integration does not double-debit when it supplies its own policy.

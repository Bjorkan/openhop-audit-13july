# BUG-070 — Identity addresses hash the public key instead of using its prefix

## TL;DR

The public identity API reports `SHA256(public_key)[0]` as an address, while MeshCore addresses and path hashes use leading public-key bytes. Routing mostly uses the correct public-key prefixes elsewhere, but logs, APIs and identity-management displays can identify a node with a different byte than the wire protocol.

**Current status: 🔴 Not fixed.** Confirmed in the supplied Core snapshot [`9ea7269`](https://github.com/openhop-dev/openhop_core/commit/9ea7269a7e7e903fe433b1f952a4026fe3dcc81b) and Repeater snapshot [`6aafa7f`](https://github.com/openhop-dev/openhop_repeater/commit/6aafa7fe991b5b3199b18149f84417f8522d94b2), as applicable. The full test suites pass, but no focused implementation or regression test resolves this compatibility difference.

## What happens

The public identity API reports `SHA256(public_key)[0]` as an address, while MeshCore addresses and path hashes use leading public-key bytes. Routing mostly uses the correct public-key prefixes elsewhere, but logs, APIs and identity-management displays can identify a node with a different byte than the wire protocol.

## How official MeshCore handles it

`Identity::getAddress()` copies the requested leading bytes of the public key directly.

## How the OpenHop stack handles it

Both local and modem identity classes hash the key before returning one byte. Repeater-facing code consumes that API for displayed or persisted address values.

## What needs to change

Return the requested leading public-key bytes. If a diagnostic SHA-256 fingerprint is desired, expose it as a separately named API to avoid conflating fingerprints with MeshCore addresses.

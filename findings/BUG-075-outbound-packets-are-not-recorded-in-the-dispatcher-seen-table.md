# BUG-075 — Outbound packets are not recorded in the dispatcher seen table

[← Active finding list](../README.md#active-findings)

## TL;DR

Core records packet hashes only after receiving them. It does not mark a packet as seen when submitting it for transmission, so a repeated copy can return and reach receive handlers. The generic own-packet heuristic cannot safely recognize every payload type.

**Current status: 🔴 Not fixed.** Rechecked against current Core head [`41b6201`](https://github.com/openhop-dev/openhop_core/commit/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8) and Repeater head [`dd6dfce`](https://github.com/openhop-dev/openhop_repeater/commit/dd6dfce9e89fab76967d91e202d8e47217c30474), as applicable. The mismatch remains present; the new commit range does not complete this MeshCore compatibility path.

## What happens

Core records packet hashes only after receiving them. It does not mark a packet as seen when submitting it for transmission, so a repeated copy can return and reach receive handlers. The generic own-packet heuristic cannot safely recognize every payload type.

## How official MeshCore handles it

Every MeshCore send path calls the packet table's `hasSeen(packet)` before queueing the packet, specifically to suppress a rebroadcast that comes back to the sender.

## How the OpenHop stack handles it

`send_packet()` serializes and transmits without inserting the exact packet hash into `PacketFilter`. Receive-side insertion happens only after a loopback has already arrived.

## What needs to change

Atomically record the exact outbound packet hash when the send is accepted, before another task can process its loopback. Coordinate this with group-message caches and failed-send semantics.

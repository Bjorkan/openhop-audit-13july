# BUG-125 — The TCP frame server emits unsolicited current-time responses as heartbeats

## TL;DR

Every 15 seconds of outbound inactivity, the TCP frame writer fabricates a RESP_CODE_CURR_TIME frame even when the client never sent CMD_GET_DEVICE_TIME. In the official companion protocol that response code is request-bound. A client that assumes command/response ordering can consume the heartbeat as the answer to another command and lose frame synchronization.

## What happens

Every 15 seconds of outbound inactivity, the TCP frame writer fabricates a RESP_CODE_CURR_TIME frame even when the client never sent CMD_GET_DEVICE_TIME. In the official companion protocol that response code is request-bound. A client that assumes command/response ordering can consume the heartbeat as the answer to another command and lose frame synchronization.

## How official MeshCore handles it

Official companion firmware labels RESP_CODE_CURR_TIME as the reply to CMD_GET_DEVICE_TIME and emits it only inside that command handler. Connection liveness is handled by the transport, not by injecting a valid application response code.

## How the OpenHop stack handles it

The writer loop wraps Queue.get() in a heartbeat timeout. On timeout it sends RESP_CODE_CURR_TIME plus the current timestamp without an inbound command. The default interval is 15 seconds, so the protocol-visible extra frame is routine rather than exceptional.

## What needs to change

Remove the application-level heartbeat. Use TCP keepalive, WebSocket/TCP liveness, or a private framing extension negotiated outside the MeshCore response-code namespace. Add a strict request/response client test that remains synchronized across long idle periods.

# BUG-105 — Companion endpoints cannot answer peer protocol requests

## TL;DR

Official companion nodes can receive authenticated `REQ` packets from contacts and answer supported requests such as telemetry. OpenHop has a `ProtocolRequestHandler`, but neither the default `MeshNode` dispatcher nor `CompanionBridge` registers it, and `CompanionRadio`’s fallback handles only `GRP_DATA`. Requests addressed to a Python companion therefore fall through without authentication or a `RESPONSE`.

## What happens

Official companion nodes can receive authenticated `REQ` packets from contacts and answer supported requests such as telemetry. OpenHop has a `ProtocolRequestHandler`, but neither the default `MeshNode` dispatcher nor `CompanionBridge` registers it, and `CompanionRadio`’s fallback handles only `GRP_DATA`. Requests addressed to a Python companion therefore fall through without authentication or a `RESPONSE`.

## How official MeshCore handles it

`BaseChatMesh::onPeerDataRecv()` calls `onContactRequest()` for `PAYLOAD_TYPE_REQ`, builds a `RESPONSE`, and returns it by reciprocal PATH, direct route or flood. The companion firmware implements telemetry permission filtering in `MyMesh::onContactRequest()`.

## How the OpenHop stack handles it

The default handler list has no `PAYLOAD_TYPE_REQ`. `CompanionBridge._handlers` likewise omits it. `CompanionRadio` registers a packet fallback solely to process binary group data.

## What needs to change

Expose contact-request callbacks on companion endpoints, construct a `ProtocolRequestHandler` with full-contact collision-safe authentication, and register it for `PAYLOAD_TYPE_REQ`. Wire response injection through direct/flood/PATH behavior and add telemetry-request interoperability tests.

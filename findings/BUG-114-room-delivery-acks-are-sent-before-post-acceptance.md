# BUG-114 — Room delivery ACKs are sent before post acceptance

## TL;DR

The generic Core text handler schedules a delivery ACK immediately after decryption, before Repeater checks room permissions, replay state, rate limits, length handling or database success. A sender can receive a valid MeshCore ACK even when the room later rejects or fails to store the post. ## What happens

The generic Core text handler schedules a delivery ACK immediately after decryption, before Repeater checks room permissions, replay state, rate limits, length handling or database success. A sender can receive a valid MeshCore ACK even when the room later rejects or fails to store the post. Official room firmware decides whether to ACK only after role and retry processing, and read-only/invalid commands receive no ACK.

## How official MeshCore handles it

The room server calculates the possible ACK hash, then sets `send_ack` according to authenticated message type, role and retry handling. Plain posts from guest/read-only clients and unauthorized CLI commands produce no ACK; an accepted plain post enables it.

## How the OpenHop stack handles it

`TextMessageHandler` creates delayed ACK tasks before publishing metadata to Repeater. The downstream `add_post()` path can still reject the payload for rate limiting or storage failure, and currently lacks permission/replay gates entirely. There is no way for the application layer to cancel the already scheduled ACK.

## What needs to change

Introduce an application acceptance decision for server-owned identities or move room TXT processing into a server-specific handler. Generate the ACK only after the room returns an accepted/retry decision, and never ACK authorization, replay, validation, rate-limit or persistence failures. Preserve generic companion-node behavior separately.

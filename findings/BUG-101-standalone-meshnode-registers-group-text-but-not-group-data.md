## What happens

The default `PacketDispatcher` installs a handler for `GRP_TXT` but not `GRP_DATA`. Official MeshCore sends both packet types through the same authenticated channel-decryption path. A standalone `MeshNode` therefore exposes binary channel data only as an opaque fallback packet, or silently loses it when no fallback consumer is installed, instead of producing the channel-data behavior available in firmware and in the companion-specific path.

## How official MeshCore handles it

`Mesh::onRecvPacket()` groups `PAYLOAD_TYPE_GRP_DATA` and `PAYLOAD_TYPE_GRP_TXT` in the same switch branch, authenticates them against matching channel secrets, and then calls `onGroupDataRecv()`. `BaseChatMesh` validates the binary header and calls `onChannelDataRecv()`.

## How the OpenHop stack handles it

`register_default_handlers()` registers `GroupTextHandler` only. `GRP_DATA` reaches the generic fallback callback rather than a standard authenticated handler. `CompanionRadio` contains a special fallback for `GRP_DATA`, which confirms that the default dispatcher itself does not provide parity for other `MeshNode` users.

## What needs to change

Add a standard group-data handler to the core handler registry and register it alongside `GRP_TXT`. Reuse the same channel-hash collision search, two-byte MAC validation, AES decryption and length validation as firmware. Expose a typed channel-data callback/event and add standalone-node tests for valid, malformed and colliding channel packets.

# BUG-118 — CLI command retries can execute administrative actions again

[← Active finding list](../README.md#active-findings)

| Field | Value |
|---|---|
| Severity | **High** |
| Area | CLI replay protection |
| Affected components | OpenHop Core, OpenHop Repeater |
| Status | Confirmed from the supplied source snapshots |

## TL;DR

OpenHop executes every authenticated text payload that matches its CLI prefix list. It does not compare the sender timestamp with the ACL client watermark and cannot distinguish a legitimate retry from a new command because Core discarded that metadata. Retrying the same command can therefore repeat administrative actions such as reboot, configuration changes or maintenance operations. Official repeater and room firmware suppress command execution when the timestamp equals the previous request.

**Current status: 🔴 Not fixed.** Rechecked against latest supplied Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519), as applicable. The three new Core commits and two new Repeater commits do not remove this mismatch; the complete regression suites and focused changed-path review found no contrary behavior.

## What happens

OpenHop executes every authenticated text payload that matches its CLI prefix list. It does not compare the sender timestamp with the ACL client watermark and cannot distinguish a legitimate retry from a new command because Core discarded that metadata. Retrying the same command can therefore repeat administrative actions such as reboot, configuration changes or maintenance operations. Official repeater and room firmware suppress command execution when the timestamp equals the previous request.

## How official MeshCore handles it

Both server implementations require `sender_timestamp >= client->last_timestamp`, mark equality as `is_retry`, update the watermark, and skip `handleCommand()` on retries. Older commands are rejected.

## How the OpenHop stack handles it

Room/repeater CLI branches call `handle_command()` immediately after prefix and admin checks. They neither read nor update `ClientInfo.last_timestamp`; the packet metadata available to them contains only decoded text.

## What needs to change

Preserve the authenticated sender timestamp, serialize per-client command processing, reject older values, and treat equal timestamps as retries that do not execute the command again. Define reply/ACK behavior separately for legacy plain CLI and `CLI_DATA`, matching each firmware server.

# BUG-102 — Repeater pushes non-zero-hop and non-discovery CONTROL packets to companions

## TL;DR

The Repeater marks every CONTROL packet non-retransmittable, but then pushes every such payload to connected companion clients. Official MeshCore exposes CONTROL data to the application only for the direct, high-bit subset when the packet has reached zero hops. Routed high-bit CONTROL, low-bit CONTROL and non-direct CONTROL can therefore appear to an OpenHop companion as locally received control data even though firmware would release them without an application callback.

## What happens

The Repeater marks every CONTROL packet non-retransmittable, but then pushes every such payload to connected companion clients. Official MeshCore exposes CONTROL data to the application only for the direct, high-bit subset when the packet has reached zero hops. Routed high-bit CONTROL, low-bit CONTROL and non-direct CONTROL can therefore appear to an OpenHop companion as locally received control data even though firmware would release them without an application callback.

## How official MeshCore handles it

Before normal direct routing, MeshCore checks for direct CONTROL with bit 7 set. It invokes `onControlDataRecv()` only when `getPathHashCount() == 0`, then releases the packet. No general CONTROL switch branch delivers arbitrary CONTROL payloads to the application.

## How the OpenHop stack handles it

The special intermediate-hop branch prevents local processing for one case, but the normal CONTROL branch unconditionally calls `deliver_control_data()` after the discovery helper. The companion push has no direct-route, high-bit or zero-hop condition.

## What needs to change

Keep the do-not-retransmit policy, but gate companion delivery with the exact firmware predicate: direct route, non-empty payload, bit 7 set and zero remaining path hashes. Add tests for flood, low-bit, zero-hop high-bit and routed high-bit CONTROL packets.

## What happens

OpenHop accepts the 12-byte handler payload containing only type, attempt, timestamp and recipient prefix, then calls the send pipeline with an empty string. Official MeshCore requires the complete command frame to contain at least one text byte. A malformed or truncated app frame can therefore become a real encrypted direct message in OpenHop instead of being rejected.

## How official MeshCore handles it

The firmware branch is entered only when the complete frame length is at least 14 bytes: one command byte, 12 metadata bytes and at least one text byte. Shorter frames fall through to the generic unsupported-command response and are not transmitted.

## How the OpenHop stack handles it

The Python handler receives the frame after the command byte has been removed, but its minimum is only 12 bytes. `data[12:]` may therefore be empty, decode to `''`, and be passed to `send_text_message()`.

## What needs to change

Require at least 13 handler-data bytes before parsing, and add a regression test for a metadata-only direct-message command. Keep this separate from the maximum-length and UTF-8 findings because it is a minimum-frame validation error.

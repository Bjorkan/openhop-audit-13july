# BUG-080 — Room-server post length uses 160 characters instead of 151 wire bytes

## TL;DR

The Python room server permits 160 characters and truncates by character count, while the firmware reserves nine bytes of the 160-byte text envelope and allows at most 151 post-text bytes. ASCII posts can already exceed the firmware limit, and multibyte UTF-8 text can exceed it by much more.

**Current status: 🔴 Not fixed.** Rechecked against latest supplied Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519), as applicable. The three new Core commits and two new Repeater commits do not remove this mismatch; the complete regression suites and focused changed-path review found no contrary behavior.

## What happens

The Python room server permits 160 characters and truncates by character count, while the firmware reserves nine bytes of the 160-byte text envelope and allows at most 151 post-text bytes. ASCII posts can already exceed the firmware limit, and multibyte UTF-8 text can exceed it by much more.

## How official MeshCore handles it

The room-server firmware defines `MAX_POST_TEXT_LEN` as `160 - 9` and stores a bounded byte string before constructing the signed payload.

## How the OpenHop stack handles it

`MAX_MESSAGE_LENGTH` is 160 despite a comment acknowledging 151, and validation uses `len(str)` plus string slicing.

## What needs to change

Enforce a 151-byte UTF-8 limit, truncate only at a valid code-point boundary if truncation is retained, and assert the final signed plaintext and packet payload limits.

# BUG-083 — CMD_SEND_TELEMETRY_REQ rejects MeshCore's self-telemetry form

## TL;DR

MeshCore supports a four-byte command frame—command plus three reserved bytes—to request the companion node's own telemetry. OpenHop requires the reserved bytes plus a 32-byte remote public key, so the valid self form is rejected with ILLEGAL_ARG and never produces a telemetry push.

## What happens

MeshCore supports a four-byte command frame—command plus three reserved bytes—to request the companion node's own telemetry. OpenHop requires the reserved bytes plus a 32-byte remote public key, so the valid self form is rejected with ILLEGAL_ARG and never produces a telemetry push.

## How official MeshCore handles it

When the frame contains only the command and three reserved bytes, firmware gathers local CayenneLPP data and immediately emits PUSH_CODE_TELEMETRY_RESPONSE with its own six-byte public-key prefix.

## How the OpenHop stack handles it

The handler has only the remote-contact branch and rejects any payload shorter than 35 bytes after the command byte is stripped.

## What needs to change

Add the three-reserved-byte self branch before remote validation. Obtain local telemetry from a bridge capability, serialize the same push layout, and keep the existing remote request path for 35-byte payloads.

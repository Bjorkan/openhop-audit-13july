# BUG-033 — Valid zero bytes are stripped from stored paths

[← Bug list](../README.md#bug-list)

| Field | Value |
|---|---|
| Severity | **High** |
| Area | Companion contact storage |
| Affected components | OpenHop Core |
| Status | Confirmed from the supplied source snapshots |

## TL;DR

Routing hashes are arbitrary bytes and can end in zero. OpenHop strips all trailing zero bytes from the fixed-width path field, shortening valid paths and making path_len disagree with the stored bytes. To fix it, calculate the exact path byte count from the unsigned encoded path_len and copy that many bytes verbatim. Ignore fixed-field padding only after the computed length.

## What happens

Routing hashes are arbitrary bytes and can end in zero. OpenHop strips all trailing zero bytes from the fixed-width path field, shortening valid paths and making path_len disagree with the stored bytes.

## How official MeshCore handles it

The companion copies the path buffer and uses encoded out_path_len to determine the meaningful byte count. Zero is not padding inside that count.

## How the OpenHop stack handles it

**OpenHop Core:** _cmd_add_update_contact applies rstrip(b"\x00") to both full and short path slices.

## What needs to change

Calculate the exact path byte count from the unsigned encoded path_len and copy that many bytes verbatim. Ignore fixed-field padding only after the computed length.

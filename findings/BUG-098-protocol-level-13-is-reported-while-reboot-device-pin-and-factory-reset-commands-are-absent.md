# BUG-098 — Protocol level 13 is reported while reboot, device-PIN and factory-reset commands are absent

## TL;DR

OpenHop reports companion protocol level 13 and exports command constants for REBOOT, SET_DEVICE_PIN and FACTORY_RESET, but none of the three is registered or implemented. A companion app that enables features from the reported protocol level receives UNSUPPORTED_CMD instead of the official behavior.

**Current status: 🔴 Not fixed.** Rechecked against current Core head [`41b6201`](https://github.com/openhop-dev/openhop_core/commit/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8) and Repeater head [`dd6dfce`](https://github.com/openhop-dev/openhop_repeater/commit/dd6dfce9e89fab76967d91e202d8e47217c30474), as applicable. The mismatch remains present; the new commit range does not complete this MeshCore compatibility path.

## What happens

OpenHop reports companion protocol level 13 and exports command constants for REBOOT, SET_DEVICE_PIN and FACTORY_RESET, but none of the three is registered or implemented. A companion app that enables features from the reported protocol level receives UNSUPPORTED_CMD instead of the official behavior.

## How official MeshCore handles it

MeshCore implements password-gated reboot, validates and persists a zero or six-digit device PIN, and password-gates filesystem format plus reboot. These branches are part of the same companion implementation that reports protocol level 13.

## How the OpenHop stack handles it

`FIRMWARE_VER_CODE` is 13 and all three command IDs are public constants, but the frame-server handler map contains no entries for command codes 19, 37 or 51. There are no policy callbacks that could provide safe host equivalents.

## What needs to change

Either implement explicit host adapters with authorization and storage semantics, or advertise a capability/protocol level that does not imply these commands. Destructive operations must be opt-in and policy-controlled; they must not be mapped blindly to process exit or arbitrary filesystem deletion.

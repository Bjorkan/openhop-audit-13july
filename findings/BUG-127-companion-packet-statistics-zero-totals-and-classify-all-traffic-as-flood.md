## TL;DR

Standalone Core records total_tx and total_rx, but the frame encoder asks for sent and recv, so the first two packet-stat fields are always zero. Repeater supplies totals but maps every sent and received packet to flood and uses policy drops as radio receive errors. ## What happens

Standalone Core records total_tx and total_rx, but the frame encoder asks for sent and recv, so the first two packet-stat fields are always zero. Repeater supplies totals but maps every sent and received packet to flood and uses policy drops as radio receive errors. Official MeshCore reports radio totals, separate flood/direct counters, and actual radio receive errors.

## How official MeshCore handles it

CMD_GET_STATS(STATS_TYPE_PACKETS) reads total packets from the radio driver, routing counters from Dispatcher, and receive-error count from the radio driver. Flood and direct values are serialized separately.

## How the OpenHop stack handles it

StatsCollector.get_totals() returns total_tx, total_rx, and tx_errors, while _cmd_get_stats() looks for sent, recv, and recv_errors. The Repeater override then assigns forwarded_count/rx_count entirely to flood and dropped_count to recv_errors, ignoring its own sent_direct_count and recv_direct_count counters.

## What needs to change

Define one canonical statistics schema shared by collector, frame encoder, and Repeater. Track actual RX errors separately from policy drops, use the existing flood/direct engine counters, and source total radio send/receive counts from the radio backend where available. Add exact seven-field wire-vector tests for standalone Core and Repeater.

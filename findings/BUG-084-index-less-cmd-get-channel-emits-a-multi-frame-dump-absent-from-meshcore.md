# BUG-084 — Index-less CMD_GET_CHANNEL emits a multi-frame dump absent from MeshCore

## TL;DR

A GET_CHANNEL command with no index causes OpenHop to emit one CHANNEL_INFO frame for every channel slot, usually 40 frames. Official MeshCore requires an index and does not define a bulk form. Strict request/response clients can be desynchronized by the unsolicited frame sequence.

**Current status: 🔴 Not fixed.** Confirmed in the supplied Core snapshot [`9ea7269`](https://github.com/openhop-dev/openhop_core/commit/9ea7269a7e7e903fe433b1f952a4026fe3dcc81b) and Repeater snapshot [`6aafa7f`](https://github.com/openhop-dev/openhop_repeater/commit/6aafa7fe991b5b3199b18149f84417f8522d94b2), as applicable. The complete existing test suites pass, but no implementation or regression test covers this MeshCore compatibility path.

## What happens

A GET_CHANNEL command with no index causes OpenHop to emit one CHANNEL_INFO frame for every channel slot, usually 40 frames. Official MeshCore requires an index and does not define a bulk form. Strict request/response clients can be desynchronized by the unsolicited frame sequence.

## How official MeshCore handles it

Firmware handles GET_CHANNEL only when the command frame includes an index and emits exactly one channel response or one NOT_FOUND error.

## How the OpenHop stack handles it

The handler treats missing data as a private `get_full_list` extension and writes every channel, including empty slots.

## What needs to change

Reject an index-less command as unsupported/illegal or move bulk enumeration to a separately negotiated command ID. Preserve one response per standard GET_CHANNEL request.

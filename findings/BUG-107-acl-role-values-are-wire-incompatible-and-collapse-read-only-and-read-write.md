# BUG-107 — ACL role values are wire-incompatible and collapse read-only and read-write

## TL;DR

OpenHop assigns role values `guest=1`, `read_write=1` and `admin=2`. MeshCore defines `guest=0`, `read_only=1`, `read_write=2` and `admin=3`. OpenHop therefore cannot represent read-only and read-write separately, emits incompatible permission bytes in login/access-list responses, and treats bit `0x02` as an admin indicator even though that bit is the official read-write role.

## What happens

OpenHop assigns role values `guest=1`, `read_write=1` and `admin=2`. MeshCore defines `guest=0`, `read_only=1`, `read_write=2` and `admin=3`. OpenHop therefore cannot represent read-only and read-write separately, emits incompatible permission bytes in login/access-list responses, and treats bit `0x02` as an admin indicator even though that bit is the official read-write role.

## How official MeshCore handles it

`ClientACL.h` reserves the low two bits as a four-value role enum and considers only masked value 3 to be admin. Room login assigns role 2 for the guest/public password and role 0 for anonymous read-only fallback.

## How the OpenHop stack handles it

The ACL constants collapse two roles and shift the others. `LoginServerHandler` sets `is_admin` whenever permission bit `0x02` is present, rather than comparing the masked role. The same hard-coded bit is used by CLI permission checks and access-list serialization.

## What needs to change

Adopt the official four role values and centralize role predicates. Change `is_admin` to masked equality, implement distinct read-only/read-write checks, update login/access-list/UI serialization and migrate any stored/configured permission values. Add fixed wire-vector tests for all four roles.

# BUG-109 — Room-server read-only clients can create posts

[← Active finding list](../README.md#active-findings)

## TL;DR

After a room client authenticates, OpenHop stores every non-prefix-classified text message as a post without checking the client’s ACL role. A client admitted through `allow_read_only` can therefore publish to the room. Official MeshCore explicitly rejects plain posts from the guest/read-only role and sends no delivery ACK.

**Current status: 🔴 Not fixed.** Rechecked against latest supplied Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519), as applicable. The three new Core commits and two new Repeater commits do not remove this mismatch; the complete regression suites and focused changed-path review found no contrary behavior.

## What happens

After a room client authenticates, OpenHop stores every non-prefix-classified text message as a post without checking the client’s ACL role. A client admitted through `allow_read_only` can therefore publish to the room. Official MeshCore explicitly rejects plain posts from the guest/read-only role and sends no delivery ACK.

## How official MeshCore handles it

In the room’s plain-text branch, firmware masks the client role and, for `PERM_ACL_GUEST`, creates neither post nor ACK. Only read-write/admin clients reach `addPost()`.

## How the OpenHop stack handles it

The room branch resolves the client but uses it only to obtain the full public key. It calls `room_server.add_post()` for every authenticated non-CLI message; there is no permissions or guest check in this path.

## What needs to change

Check the official masked role before accepting a plain post. Reject read-only/guest clients before storage, rate-limit accounting and ACK generation. Add tests for guest, read-only, read-write and admin roles after correcting the role encoding.

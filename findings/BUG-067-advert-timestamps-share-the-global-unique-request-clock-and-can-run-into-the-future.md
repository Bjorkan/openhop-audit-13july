# BUG-067 — Advert timestamps share the global unique-request clock and can run into the future

[← Active finding list](../README.md#active-findings)

| Field | Value |
|---|---|

| Area | Advertisement timestamps |

## TL;DR

OpenHop uses one global strictly increasing timestamp generator for adverts and replay-sensitive request packets. A burst of requests within one second advances the counter beyond wall time; the next advert can then carry a future timestamp, causing peers to reject later normal adverts until their clocks catch up.

**Current status: 🔴 Not fixed.** Confirmed in the supplied Core snapshot [`9ea7269`](https://github.com/openhop-dev/openhop_core/commit/9ea7269a7e7e903fe433b1f952a4026fe3dcc81b) and Repeater snapshot [`6aafa7f`](https://github.com/openhop-dev/openhop_repeater/commit/6aafa7fe991b5b3199b18149f84417f8522d94b2), as applicable. The full test suites pass, but no focused implementation or regression test resolves this compatibility difference.

## What happens

OpenHop uses one global strictly increasing timestamp generator for adverts and replay-sensitive request packets. A burst of requests within one second advances the counter beyond wall time; the next advert can then carry a future timestamp, causing peers to reject later normal adverts until their clocks catch up.

## How official MeshCore handles it

MeshCore creates adverts with the ordinary current time. It reserves the unique monotonic timestamp helper for request/login/replay-protected operations that require uniqueness.

## How the OpenHop stack handles it

`PacketBuilder.create_advert()` calls `_get_timestamp()`, the same process-global counter used by request builders. The counter is also shared by every identity in the Python process.

## What needs to change

Use current wall/RTC time for adverts. Keep a separate unique request timestamp source, preferably scoped per local identity or protocol session, and add burst tests that ensure adverts never jump ahead because of unrelated requests.

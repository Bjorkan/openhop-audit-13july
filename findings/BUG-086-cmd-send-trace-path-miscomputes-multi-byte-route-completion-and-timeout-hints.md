# BUG-086 — CMD_SEND_TRACE_PATH miscomputes multi-byte route completion and timeout hints

[← Active finding list](../README.md#active-findings)

| Area | TRACE routing |

## TL;DR

OpenHop validates multi-byte TRACE paths, but its timeout grows with the number of path bytes rather than hop count and airtime. Its synthetic final-hop shortcut compares only the last byte to a one-byte local hash, which is incorrect for 2-, 4-, and 8-byte path hashes and can fire on a suffix collision.

**Current status: 🔴 Not fixed.** Rechecked against current Core head [`41b6201`](https://github.com/openhop-dev/openhop_core/commit/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8) and Repeater head [`dd6dfce`](https://github.com/openhop-dev/openhop_repeater/commit/dd6dfce9e89fab76967d91e202d8e47217c30474), as applicable. The mismatch remains present; the new commit range does not complete this MeshCore compatibility path.

## What happens

OpenHop validates multi-byte TRACE paths, but its timeout grows with the number of path bytes rather than hop count and airtime. Its synthetic final-hop shortcut compares only the last byte to a one-byte local hash, which is incorrect for 2-, 4-, and 8-byte path hashes and can fire on a suffix collision.

## How official MeshCore handles it

MeshCore calculates the TRACE hop count as `path_len >> path_sz`, includes packet airtime in the direct timeout, and uses `Identity::isHashMatch()` over the complete selected hash width when forwarding each hop.

## How the OpenHop stack handles it

The command computes `base + path_len * per_byte`, then decides it is the destination from `path_bytes[-1] == local_hash`. The actual TRACE receive pipeline is bypassed for this immediate push.

## What needs to change

Calculate timeout from packet airtime and decoded hop count. Remove the one-byte synthetic destination check or compare the full final hash with the configured width through the identity API, and let the normal TRACE receive path produce completion data.

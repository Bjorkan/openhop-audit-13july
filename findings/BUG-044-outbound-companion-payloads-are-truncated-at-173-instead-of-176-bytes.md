# BUG-044 — Outbound companion payloads are truncated at 173 instead of 176 bytes

[← Bug list](../README.md#bug-list)

| Field | Value |
|---|---|

| Area | Companion framing |
| Affected components | OpenHop Core |
| Status | Confirmed from the supplied source snapshots |

## TL;DR

MeshCore defines MAX_FRAME_SIZE as the payload passed to writeFrame; its three-byte serial prefix is additional. OpenHop subtracts that prefix from 176 and truncates command payloads at 173. To fix it, treat 176 as the framed payload maximum on input and output. Allocate/write 179 total bytes including prefix and length, and reject rather than silently truncate responses whose command contract cannot be shortened.

## What happens

MeshCore defines MAX_FRAME_SIZE as the payload passed to writeFrame; its three-byte serial prefix is additional. OpenHop subtracts that prefix from 176 and truncates command payloads at 173.

## How official MeshCore handles it

ArduinoSerialInterface::writeFrame accepts len <= 176, then writes the three-byte prefix and all len payload bytes.

## How the OpenHop stack handles it

**OpenHop Core:** MAX_PAYLOAD_SIZE is defined as MAX_FRAME_SIZE - 3. _enqueue_frame truncates data to that smaller value before adding its own prefix.

## What needs to change

Treat 176 as the framed payload maximum on input and output. Allocate/write 179 total bytes including prefix and length, and reject rather than silently truncate responses whose command contract cannot be shortened.

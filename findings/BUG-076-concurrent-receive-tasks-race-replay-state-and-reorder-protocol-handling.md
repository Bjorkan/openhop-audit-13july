# BUG-076 — Concurrent receive tasks race replay state and reorder protocol handling

[← Active finding list](../README.md#active-findings)

| Field | Value |
|---|---|
| Severity | **High** |
| Area | Receive scheduling and REQ replay protection |
| Affected components | OpenHop Core |
| Status | Confirmed from the supplied source snapshots |

## TL;DR

OpenHop starts an independent asyncio task for every received radio frame. The packet hash check itself does not yield and is therefore not the race: the concrete defect is that authenticated protocol requests read a client's replay watermark, await command handling, and only then advance the watermark. Two concurrent REQs for the same client can both pass the same old watermark. Their completion order can also move `last_timestamp` backwards, allowing an older request to become acceptable again.

The mismatch remains present; the new commit range does not complete this MeshCore compatibility path.

## What happens

`Dispatcher._on_packet_received()` schedules `_process_received_packet()` with `create_task()` for every radio callback. Those tasks can reach `ProtocolRequestHandler` concurrently.

For a valid authenticated REQ, the handler:

1. reads `last_timestamp`;
2. rejects only when the incoming timestamp is not newer;
3. awaits `_handle_request()`;
4. writes the incoming timestamp as the new watermark.

The await splits the check from the state update. For example, requests with timestamps 101 and 102 can both read 100 and both be accepted. If 102 finishes first and 101 finishes later, the stored watermark becomes 101. Timestamp 102 can then be replayed and accepted again. Side effects from the two commands can also occur in an order different from radio receive order.

The nearby duplicate lookup followed by `track_packet()` does not contain an await and is not, by itself, an asyncio race. This finding is specifically about stateful handlers that yield after validating mutable protocol state.

## How official MeshCore handles it

The firmware dispatcher drains received packets from one dispatcher loop. Both immediately processed direct packets and delayed inbound flood packets ultimately enter `processRecvPacket()` serially. A request's replay comparison, command handling and watermark advancement therefore cannot overlap another packet on the same dispatcher execution path.

## How the OpenHop stack handles it

OpenHop deliberately decouples the radio callback by creating one task per frame, but it provides no serial receive worker and no per-client lock or reservation around REQ replay state. `_advance_client_watermark()` also assigns the supplied timestamp directly rather than enforcing monotonicity as a defensive invariant.

## What needs to change

Either process parsed inbound frames through one bounded serial receive worker, matching the firmware ordering model, or protect each authenticated client's replay check, command execution and watermark update with one per-client lock. The stored watermark must never decrease. Add tests that:

- overlap two valid REQs for the same client;
- force the newer request to finish first;
- verify each request is executed at most once;
- verify the final watermark is the maximum accepted timestamp;
- verify replay of either accepted timestamp is rejected.

# BUG-068 — Timed-out dispatcher ACK waiters are never removed

## TL;DR

Every unique ACK CRC registered by `wait_for_ack()` remains in `_waiting_acks` after a timeout. Only an eventual matching ACK removes it. Repeated failed transmissions therefore grow the waiter dictionary indefinitely and retain asyncio event objects.

**Current status: 🔴 Not fixed.** Rechecked against current Core head [`41b6201`](https://github.com/openhop-dev/openhop_core/commit/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8) and Repeater head [`dd6dfce`](https://github.com/openhop-dev/openhop_repeater/commit/dd6dfce9e89fab76967d91e202d8e47217c30474), as applicable. ## What happens

Every unique ACK CRC registered by `wait_for_ack()` remains in `_waiting_acks` after a timeout. Only an eventual matching ACK removes it. Repeated failed transmissions therefore grow the waiter dictionary indefinitely and retain asyncio event objects.

## How official MeshCore handles it

MeshCore's pending outbound/ACK state is tied to bounded queues and request lifetimes. Expired sends do not leave an unbounded registry of waiter objects.

## How the OpenHop stack handles it

The timeout branch logs and returns false without popping the CRC. Maintenance cleans `_recent_acks`, not `_waiting_acks`.

## What needs to change

Remove the exact event in a `finally` block when it is still the registered waiter. Preserve race safety so a replacement waiter for the same CRC is not removed.

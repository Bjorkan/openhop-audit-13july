# BUG-110 — Room-server message replays can create duplicate posts

[← Active finding list](../README.md#active-findings)

## TL;DR

OpenHop discards the sender timestamp before room handling, substitutes the server’s current wall time, and inserts every accepted plain message. After the generic packet seen-cache expires—or when retry bits produce a different packet hash—the same signed/encrypted logical message can create another room post. MeshCore tracks each client’s sender timestamp and treats equality as a retry that may be ACKed but is never inserted again.

## What happens

OpenHop discards the sender timestamp before room handling, substitutes the server’s current wall time, and inserts every accepted plain message. After the generic packet seen-cache expires—or when retry bits produce a different packet hash—the same signed/encrypted logical message can create another room post. MeshCore tracks each client’s sender timestamp and treats equality as a retry that may be ACKed but is never inserted again.

## How official MeshCore handles it

The room parser accepts only sender timestamps greater than or equal to `client->last_timestamp`, marks equality as `is_retry`, advances the watermark, and calls `addPost()` only when `!is_retry`.

## How the OpenHop stack handles it

The core handler exposes only `{'text': decoded_msg}` on the packet. Repeater room handling sets `sender_timestamp = int(time.time())` rather than using the decrypted timestamp, and `add_post()` always inserts if its independent rate limit allows it. ACL `last_timestamp` is not consulted.

## What needs to change

Preserve the authenticated sender timestamp and text type in the packet/event passed to Repeater. Apply a per-client timestamp watermark before command/post processing, reject older values, treat equality as a retry, and only ACK—not reinsert—eligible retries. Serialize concurrent messages per client.

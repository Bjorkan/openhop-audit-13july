# BUG-082 — CMD_SET_RADIO_TX_POWER ignores the advertised hardware limit

## TL;DR

SELF_INFO can advertise a backend-specific maximum TX power, but CMD_SET_RADIO_TX_POWER validates against a separate hardcoded range of -9 through 29 dBm. A backend that advertises 14 dBm can acknowledge and apply 22 dBm, while a backend capable of 30 dBm or more is rejected despite its advertised capability.

**Current status: 🟡 Partially fixed.** SELF_INFO now resolves and advertises the real backend maximum TX power through the new radio-capability layer, including the SX1262 22 dBm capability and Repeater host wiring. The command handler still validates only the generic signed range −9 through 29 dBm and never compares the requested value with `get_max_tx_power_dbm()`, so a request above the advertised hardware maximum can still be accepted; see [`commands_device.py` L345–L369](https://github.com/openhop-dev/openhop_core/blob/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8/src/openhop_core/companion/frame_server/commands_device.py#L345-L369) and [`radio_capabilities.py`](https://github.com/openhop-dev/openhop_core/blob/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8/src/openhop_core/companion/radio_capabilities.py). Reviewed at Core head [`41b6201`](https://github.com/openhop-dev/openhop_core/commit/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8) and Repeater head [`dd6dfce`](https://github.com/openhop-dev/openhop_repeater/commit/dd6dfce9e89fab76967d91e202d8e47217c30474).

## What happens

SELF_INFO can advertise a backend-specific maximum TX power, but CMD_SET_RADIO_TX_POWER validates against a separate hardcoded range of -9 through 29 dBm. A backend that advertises 14 dBm can acknowledge and apply 22 dBm, while a backend capable of 30 dBm or more is rejected despite its advertised capability.

## How official MeshCore handles it

MeshCore validates the requested signed power against `MAX_LORA_TX_POWER`, the same target-specific limit reported to the companion app.

## How the OpenHop stack handles it

The command never calls `get_max_tx_power_dbm()`. It accepts every value below 30 after the -9 dBm floor, independently of the capability reported by SELF_INFO.

## What needs to change

Use the same hardware capability getter for both SELF_INFO and command validation. Clamp nothing silently: reject values below the minimum or above the advertised maximum and test backends with lower and higher limits.

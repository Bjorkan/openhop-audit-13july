# BUG-100 — The global room-server rate limiter releases its lock before transmission begins

[← Active finding list](../README.md#active-findings)

| Field | Value |
|---|---|
| Severity | **High** |
| Area | Room server scheduling |
| Affected components | OpenHop Repeater |
| Status | Confirmed from the supplied source snapshots |

## TL;DR

`GlobalRateLimiter.acquire()` enters and exits `async with self.lock` before returning. The caller's radio transmission and ACK wait therefore occur with no global limiter lock held. Multiple room-server loops can begin pushes concurrently, bypass the intended 1.1-second inter-message gap, and create multiple in-flight room deliveries even though the code comments claim only one radio message is active.

**Current status: 🔴 Not fixed.** Rechecked against current Core head [`41b6201`](https://github.com/openhop-dev/openhop_core/commit/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8) and Repeater head [`dd6dfce`](https://github.com/openhop-dev/openhop_repeater/commit/dd6dfce9e89fab76967d91e202d8e47217c30474), as applicable. The mismatch remains present; the new commit range does not complete this MeshCore compatibility path.

## What happens

`GlobalRateLimiter.acquire()` enters and exits `async with self.lock` before returning. The caller's radio transmission and ACK wait therefore occur with no global limiter lock held. Multiple room-server loops can begin pushes concurrently, bypass the intended 1.1-second inter-message gap, and create multiple in-flight room deliveries even though the code comments claim only one radio message is active.

## How official MeshCore handles it

The firmware room server has one loop, selects at most one client/post per iteration, and schedules the next successful push after `SYNC_PUSH_INTERVAL` (1200 ms). It does not permit another room push to overlap that scheduling slot.

## How the OpenHop stack handles it

Every configured room has an independent async sync loop and all rooms share a limiter object. However, `acquire()` uses a context manager internally and returns only after that context has released the lock; `release()` merely updates a timestamp. The critical section described in comments does not exist.

## What needs to change

Use a real acquire/release pair or an async context manager that remains held across packet injection and the intended serialization interval. Put release in `finally`, use a monotonic clock, and test concurrent pushes from two room instances plus exceptions and early-return paths.

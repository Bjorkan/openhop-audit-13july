## TL;DR

The shortest official RAW_DATA body is five bytes after the command: encoded path length `0`, followed by a four-byte payload. OpenHop performs an initial six-byte minimum check before its otherwise-correct path-aware check, so that valid zero-hop frame is rejected as unsupported.

**Current status: 🔴 Not fixed.** Rechecked against latest supplied Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519), as applicable. ## What happens

The shortest official RAW_DATA body is five bytes after the command: encoded path length `0`, followed by a four-byte payload. OpenHop performs an initial six-byte minimum check before its otherwise-correct path-aware check, so that valid zero-hop frame is rejected as unsupported.

## How official MeshCore handles it

Firmware accepts complete frames of six bytes. With `path_len = 0`, the remaining four bytes satisfy the documented minimum payload and are sent directly with an empty route.

## How the OpenHop stack handles it

The first guard requires six bytes after the command was stripped. The later expression `1 + path_byte_len + 4` correctly computes five bytes for zero-hop, but it is unreachable for that valid minimum frame.

## What needs to change

Remove the redundant fixed guard or reduce it to five bytes and rely on the encoded-path-aware bounds check. Add zero-hop, one-hop and truncated-path tests.

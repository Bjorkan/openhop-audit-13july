# BUG-112 — Unsupported inbound text types are delivered as application messages

[← Active finding list](../README.md#active-findings)

## TL;DR

OpenHop extracts a six-bit `txt_type` but does not reject values outside plain, CLI data and signed plain. Unsupported/reserved types are decoded, emitted through the normal message event and stored in `packet.decrypted`. Official MeshCore logs the unsupported type and makes no application callback.

**Current status: 🔴 Not fixed.** Rechecked against latest supplied Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519), as applicable. The three new Core commits and two new Repeater commits do not remove this mismatch; the complete regression suites and focused changed-path review found no contrary behavior.

## What happens

OpenHop extracts a six-bit `txt_type` but does not reject values outside plain, CLI data and signed plain. Unsupported/reserved types are decoded, emitted through the normal message event and stored in `packet.decrypted`. Official MeshCore logs the unsupported type and makes no application callback.

## How official MeshCore handles it

`BaseChatMesh` has explicit branches for `TXT_TYPE_PLAIN`, `TXT_TYPE_CLI_DATA` and `TXT_TYPE_SIGNED_PLAIN`; every other flag value reaches a terminal unsupported-type branch.

## How the OpenHop stack handles it

The handler changes ACK behavior according to known types, but all types continue into visible UTF-8 decoding and message publication unless a signed frame is too short. This lets reserved wire values masquerade as ordinary received messages.

## What needs to change

Validate `txt_type` immediately after authenticated decryption. Consume unsupported packets without ACK, command capture, event publication or contact activity updates. Add one test for every supported type plus representative reserved values.

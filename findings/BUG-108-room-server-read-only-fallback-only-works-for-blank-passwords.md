## TL;DR

With `allow_read_only` enabled, official room firmware accepts a client as read-only after any admin/guest password mismatch. OpenHop considers read-only only inside the blank-password branch; a non-empty wrong password is rejected. It also rejects the room outright when neither password is configured, even though firmware can still operate as a read-only room.

## What happens

With `allow_read_only` enabled, official room firmware accepts a client as read-only after any admin/guest password mismatch. OpenHop considers read-only only inside the blank-password branch; a non-empty wrong password is rejected. It also rejects the room outright when neither password is configured, even though firmware can still operate as a read-only room.

## How official MeshCore handles it

Room login first checks admin and public passwords, then assigns the guest/read-only role whenever both fail and `allow_read_only` is true. The fallback is independent of whether the supplied password was blank.

## How the OpenHop stack handles it

`authenticate_client()` creates a read-only guest only under `if not password`. The non-empty mismatch branch returns failure immediately. Before that, room identities with no admin or guest password configured are rejected without consulting `allow_read_only`.

## What needs to change

For room identities, evaluate admin and guest passwords first and then apply the configured read-only fallback to any mismatch, including blank and non-blank values. Permit a deliberately passwordless read-only room. Keep stricter repeater login policy separate and add behavior tests for all combinations.

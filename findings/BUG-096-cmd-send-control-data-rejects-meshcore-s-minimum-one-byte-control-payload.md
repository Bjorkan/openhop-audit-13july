# BUG-096 — CMD_SEND_CONTROL_DATA rejects MeshCore's minimum one-byte control payload

## TL;DR

Official MeshCore accepts a control body consisting of one byte when its high bit is set and sends it zero-hop. OpenHop requires at least two body bytes, so a valid minimum control packet such as `0x80` is rejected with ILLEGAL_ARG and never reaches the mesh.

**Current status: 🔴 Not fixed.** Confirmed in the supplied Core snapshot [`9ea7269`](https://github.com/openhop-dev/openhop_core/commit/9ea7269a7e7e903fe433b1f952a4026fe3dcc81b) and Repeater snapshot [`6aafa7f`](https://github.com/openhop-dev/openhop_repeater/commit/6aafa7fe991b5b3199b18149f84417f8522d94b2), as applicable. The complete existing test suites pass, but no implementation or regression test covers this MeshCore compatibility path.

## What happens

Official MeshCore accepts a control body consisting of one byte when its high bit is set and sends it zero-hop. OpenHop requires at least two body bytes, so a valid minimum control packet such as `0x80` is rejected with ILLEGAL_ARG and never reaches the mesh.

## How official MeshCore handles it

The complete firmware frame requires only two bytes: the command byte and one control byte. It passes all bytes after the command directly to `createControlData()` and sends the resulting packet zero-hop.

## How the OpenHop stack handles it

The Python command byte has already been removed, yet `_cmd_send_control_data()` still requires `len(data) >= 2`. This creates an off-by-one minimum and rejects the official one-byte body.

## What needs to change

Lower the handler-data minimum to one byte while preserving the high-bit validation. Test one-byte control data, discovery payloads with tags, and allocation failure response mapping.

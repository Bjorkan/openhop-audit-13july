# BUG-078 — Companion node names are limited by characters instead of the 31-byte field

## TL;DR

OpenHop truncates node names to 31 Python characters. UTF-8 names can therefore occupy far more than the firmware's 31-byte field and produce oversized or inconsistent SELF_INFO and advert data.

**Current status: 🔴 Not fixed.** Confirmed in the supplied Core snapshot [`9ea7269`](https://github.com/openhop-dev/openhop_core/commit/9ea7269a7e7e903fe433b1f952a4026fe3dcc81b) and Repeater snapshot [`6aafa7f`](https://github.com/openhop-dev/openhop_repeater/commit/6aafa7fe991b5b3199b18149f84417f8522d94b2), as applicable. The full test suites pass, but no focused implementation or regression test resolves this compatibility difference.

## What happens

OpenHop truncates node names to 31 Python characters. UTF-8 names can therefore occupy far more than the firmware's 31-byte field and produce oversized or inconsistent SELF_INFO and advert data.

## How official MeshCore handles it

SET_ADVERT_NAME copies at most `sizeof(node_name)-1` input bytes and terminates the fixed byte buffer.

## How the OpenHop stack handles it

The preference setter slices the Unicode string before encoding. APP_START later appends the complete UTF-8 byte sequence.

## What needs to change

Normalize at the UTF-8 byte boundary, truncate without splitting a code point, and store the resulting canonical string. Test 1–4-byte code points around the 31-byte boundary.

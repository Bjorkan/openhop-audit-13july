# BUG-093 — The repeater does not implement MeshCore telemetry REQ responses

## TL;DR

OpenHop Core defines REQ_TYPE_GET_TELEMETRY_DATA and companions can transmit it, but the Repeater helper never registers a handler for request type 0x03. An authenticated telemetry request is consumed without a response, whereas the official repeater returns CayenneLPP base and permitted sensor data.

**Current status: 🔴 Not fixed.** Rechecked against current Core head [`41b6201`](https://github.com/openhop-dev/openhop_core/commit/41b6201ea2e3cb9b8468b0eb80c9e22fdad4a6c8) and Repeater head [`dd6dfce`](https://github.com/openhop-dev/openhop_repeater/commit/dd6dfce9e89fab76967d91e202d8e47217c30474), as applicable. The mismatch remains present; the new commit range does not complete this MeshCore compatibility path.

## What happens

OpenHop Core defines REQ_TYPE_GET_TELEMETRY_DATA and companions can transmit it, but the Repeater helper never registers a handler for request type 0x03. An authenticated telemetry request is consumed without a response, whereas the official repeater returns CayenneLPP base and permitted sensor data.

## How official MeshCore handles it

The simple repeater handles request type 0x03, applies the inverse permission mask (and guest restrictions), adds battery and sensor telemetry, and returns it in an authenticated RESPONSE.

## How the OpenHop stack handles it

`register_identity()` installs status, access-list, neighbours and owner-info handlers only. The Core request handler has no default telemetry implementation, so it logs no handler and returns no response.

## What needs to change

Register REQ_TYPE_GET_TELEMETRY_DATA for each repeater identity. Build CayenneLPP from the configured sensor manager, enforce the same permission mask and guest restrictions, and add end-to-end tests from companion request through encrypted RESPONSE.

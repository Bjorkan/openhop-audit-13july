# BUG-111 — Repeater and room ACL identities are lost on restart

## TL;DR

OpenHop constructs a fresh in-memory ACL for every repeater and room identity at startup. Client public keys, roles and learned direct paths are never loaded or saved. After a restart, blank-password login for a previously authorized client fails or changes role, access-list edits disappear, and room/repeater authorization no longer matches the persistent ACL used by official firmware.

**Current status: 🔴 Not fixed.** Rechecked against latest supplied Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519), as applicable. ## What happens

OpenHop constructs a fresh in-memory ACL for every repeater and room identity at startup. Client public keys, roles and learned direct paths are never loaded or saved. After a restart, blank-password login for a previously authorized client fails or changes role, access-list edits disappear, and room/repeater authorization no longer matches the persistent ACL used by official firmware.

## How official MeshCore handles it

`ClientACL::load()` reads public key, permissions, room sync watermark, path and shared-secret fields from `/s_contacts`; `save()` writes persistent non-guest entries. Shared secrets are recalculated against the current local identity while roles and routes survive reboot.

## How the OpenHop stack handles it

`ACL` owns only a Python dictionary. `LoginHelper.register_identity()` creates a new empty `ACL` object each time. SQLite persists separate room sync guard rows, but it has no canonical table/load path for ACL identities, roles and routes.

## What needs to change

Persist ACL entries per local identity, including full public key, official permissions byte, sync state and encoded route. Load them before login handlers become reachable, recalculate shared secrets, save after role/path changes, and define migration/guest-retention behavior matching firmware.

# BUG-113 — Room-server CLI and posts are classified by text prefix instead of wire type

## TL;DR

MeshCore room traffic distinguishes CLI commands from posts with the authenticated `txt_type` field. OpenHop discards that field before Repeater processing and guesses from a list of text prefixes. A legitimate plain post beginning with `set `, `get `, `reboot`, and similar words can be denied or executed as CLI, while a `CLI_DATA` frame with an unrecognized prefix can be stored as a public post.

## What happens

MeshCore room traffic distinguishes CLI commands from posts with the authenticated `txt_type` field. OpenHop discards that field before Repeater processing and guesses from a list of text prefixes. A legitimate plain post beginning with `set `, `get `, `reboot`, and similar words can be denied or executed as CLI, while a `CLI_DATA` frame with an unrecognized prefix can be stored as a public post.

## How official MeshCore handles it

The room server switches directly on `TXT_TYPE_CLI_DATA` versus `TXT_TYPE_PLAIN` after decryption and replay validation. Command text content does not decide the message class.

## How the OpenHop stack handles it

Core puts only `text` in `packet.decrypted`. Repeater calls `_is_cli_command(message_text)`, whose hard-coded prefix list controls whether the payload is executed as CLI or inserted as a post. It then hardcodes accepted posts back to `TXT_TYPE_PLAIN`.

## What needs to change

Preserve authenticated `txt_type` through Core and branch exclusively on it in room/repeater handlers. Keep any prefix recognition inside the CLI parser only after a CLI message class has been established. Reject unsupported types and test plain posts that begin with every command prefix.

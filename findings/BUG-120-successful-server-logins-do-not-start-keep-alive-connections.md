# BUG-120 — Successful server logins do not start keep-alive connections

## TL;DR

OpenHop parses the keep-alive byte from a successful server login but does not create or service a connection. It has no equivalent of MeshCore's connection table, periodic authenticated KEEP_ALIVE requests, ACK/activity refresh or expiry. A server may therefore consider the client disconnected even though OpenHop reported login success, and the advertised HAS_CONNECTION command cannot be backed by real state.

**Current status: 🔴 Not fixed.** Rechecked against latest supplied Core head [`abe1d85`](https://github.com/openhop-dev/openhop_core/commit/abe1d857734d5d1f549ada27af6f95c7e5e53e95) and Repeater head [`b62960f`](https://github.com/openhop-dev/openhop_repeater/commit/b62960fa3041447bedadb2131d892a6565d7e519), as applicable. The three new Core commits and two new Repeater commits do not remove this mismatch; the complete regression suites and focused changed-path review found no contrary behavior.

## What happens

OpenHop parses the keep-alive byte from a successful server login but does not create or service a connection. It has no equivalent of MeshCore's connection table, periodic authenticated KEEP_ALIVE requests, ACK/activity refresh or expiry. A server may therefore consider the client disconnected even though OpenHop reported login success, and the advertised HAS_CONNECTION command cannot be backed by real state.

## How official MeshCore handles it

After a successful login, companion firmware multiplies the encoded interval by 16 seconds and calls `startConnection()`. BaseChatMesh stores the server, schedules periodic KEEP_ALIVE requests, correlates their ACKs, refreshes activity on server traffic and expires silent connections.

## How the OpenHop stack handles it

`LoginResponseHandler` returns `keep_alive_interval` and updates only `last_login_success` and `is_admin`. No connection object is created and no periodic task consumes the interval.

## What needs to change

Add a bounded connection manager shared by CompanionRadio and CompanionBridge. Start it after successful login, convert the wire interval correctly, send authenticated KEEP_ALIVE requests with expected ACKs, refresh activity on valid server traffic, expire stale entries, stop them on logout, and use it for HAS_CONNECTION. Tests should cover zero intervals, ACKs, unsolicited activity, timeout and reconnect.

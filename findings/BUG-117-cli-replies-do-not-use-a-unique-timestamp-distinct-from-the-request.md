# BUG-117 — CLI replies do not use a unique timestamp distinct from the request

## TL;DR

OpenHop CLI replies use `int(time.time())`, which has one-second resolution, and never compare it with the request timestamp. Multiple equal replies in one second can produce identical encrypted packets and packet hashes, and a fast reply can reuse the request timestamp. Both official repeater and room firmware use the unique RTC clock and explicitly increment equality to keep request and reply distinct in CLI views and deduplication.

## What happens

OpenHop CLI replies use `int(time.time())`, which has one-second resolution, and never compare it with the request timestamp. Multiple equal replies in one second can produce identical encrypted packets and packet hashes, and a fast reply can reuse the request timestamp. Both official repeater and room firmware use the unique RTC clock and explicitly increment equality to keep request and reply distinct in CLI views and deduplication.

## How official MeshCore handles it

The repeater calls `getCurrentTimeUnique()` for every reply and increments the result if it equals `sender_timestamp`. The room server performs the same equality workaround before creating the reply datagram.

## How the OpenHop stack handles it

The shared `_send_cli_reply()` helper does not receive the decrypted sender timestamp. It serializes whole-second wall time directly into the reply, so neither uniqueness nor request/reply distinction is guaranteed.

## What needs to change

Carry `sender_timestamp` into the CLI handler, allocate replies through the same monotonic unique-timestamp source used by packet builders, and increment if equality remains possible. Add tests for two identical replies in one second and for a request timestamp equal to local time.

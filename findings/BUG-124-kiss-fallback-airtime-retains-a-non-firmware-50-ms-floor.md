# BUG-124 — KISS fallback airtime retains a non-firmware 50 ms floor

## TL;DR

When a KISS modem does not answer GET_AIRTIME, OpenHop's fallback estimator forces every packet to at least 50 ms even when the RadioLib-equivalent calculation is shorter. The source itself notes that firmware reports true sub-50 ms airtime. Repeater duty-cycle accounting and timing metadata can therefore overcharge short packets only on this fallback path.

## What happens

When a KISS modem does not answer GET_AIRTIME, OpenHop's fallback estimator forces every packet to at least 50 ms even when the RadioLib-equivalent calculation is shorter. The source itself notes that firmware reports true sub-50 ms airtime. Repeater duty-cycle accounting and timing metadata can therefore overcharge short packets only on this fallback path.

## How official MeshCore handles it

RadioLibWrapper returns `getTimeOnAir(len_bytes) / 1000` directly, without a minimum. The modem's normal GET_AIRTIME response therefore preserves short airtimes.

## How the OpenHop stack handles it

`PacketTimingUtils.estimate_airtime_ms()` computes the shared estimator and then returns `max(airtime_ms, 50.0)`. KISS send uses that value whenever GET_AIRTIME is unavailable.

## What needs to change

Remove the legacy floor from the physical airtime estimate. Any caller needing a scheduling safety margin should add it explicitly and separately from transmitted airtime, so duty-cycle accounting remains based on actual estimated time-on-air. Add fast-radio/short-packet vectors below 50 ms and a fallback-path integration test.
